## 三、单周期 CPU 关键代码

### 1. 单周期顶层数据通路

文件：`source-sc/SCCPU.v`

核心连接关系：

```verilog
assign Addr_out = aluout;
assign B = (ALUSrc) ? immout : RD2;
assign Data_out = RD2;

PC  U_PC(.clk(clk), .rst(reset), .NPC(NPC), .PC(PC_out));
NPC U_NPC(.PC(PC_out), .NPCOp(NPCOp), .IMM(immout), .RS1(RD1), .NPC(NPC));
EXT U_EXT(..., .EXTOp(EXTOp), .immout(immout));
RF  U_RF(..., .RFWr(RegWrite), .A1(rs1), .A2(rs2), .A3(rd), .WD(WD), ...);
alu U_alu(.A(RD1), .B(B), .ALUOp(ALUOp), .C(aluout), .Zero(Zero));
```

说明：

- `PC_out` 送入指令存储器取指。
- `ctrl.v` 根据 opcode、funct3、funct7 产生控制信号。
- `ALUSrc=0` 时 ALU 第二操作数来自寄存器 `RD2`，`ALUSrc=1` 时来自立即数 `immout`。
- `Addr_out` 是 ALU 结果，用于 `lw/sw` 的数据存储器地址。
- `Data_out=RD2`，用于 `sw` 写入内存。

写回数据选择：

```verilog
always @* begin
    case(WDSel)
        `WDSel_FromALU: WD <= aluout;
        `WDSel_FromMEM: WD <= Data_in;
        `WDSel_FromPC:  WD <= PC_out + 4;
        default:        WD <= aluout;
    endcase
end
```

说明：

- R 型、I 型算术逻辑指令写回 ALU 结果。
- `lw` 写回数据存储器读出的 `Data_in`。
- `jal` 和 `jalr` 写回 `PC + 4` 作为返回地址。

### 2. 控制器

文件：`source-sc/ctrl.v`

典型控制逻辑：

```verilog
7'b0000011: begin // lw
   RegWrite = 1'b1;
   ALUSrc   = 1'b1;
   EXTOp    = `EXT_CTRL_ITYPE;
   ALUOp    = `ALUOp_add;
   WDSel    = `WDSel_FromMEM;
end

7'b0100011: begin // sw
   MemWrite = 1'b1;
   ALUSrc   = 1'b1;
   EXTOp    = `EXT_CTRL_STYPE;
   ALUOp    = `ALUOp_add;
end
```

说明：

- `lw` 和 `sw` 都用 `rs1 + imm` 计算访存地址。
- `lw` 需要写寄存器，写回来源是内存。
- `sw` 不写寄存器，只打开数据存储器写使能。

跳转控制：

```verilog
7'b1101111: begin // jal
   RegWrite = 1'b1;
   EXTOp    = `EXT_CTRL_JTYPE;
   NPCOp    = `NPC_JUMP;
   WDSel    = `WDSel_FromPC;
end

7'b1100111: begin // jalr
   RegWrite = 1'b1;
   ALUSrc   = 1'b1;
   EXTOp    = `EXT_CTRL_ITYPE;
   ALUOp    = `ALUOp_add;
   NPCOp    = `NPC_JALR;
   WDSel    = `WDSel_FromPC;
end
```

说明：

- `jal` 的目标地址是 `PC + J-type imm`。
- `jalr` 的目标地址是 `rs1 + I-type imm`，并清零最低位。
- 两者都把 `PC + 4` 写回 `rd`，用于函数返回。

### 3. ALU 和分支判断

文件：`source-sc/alu.v`

```verilog
`ALUOp_add:  C = A + B;
`ALUOp_sub:  C = A - B;
`ALUOp_slt:  C = (A < B) ? 32'd1 : 32'd0;
`ALUOp_and:  C = A & B;
`ALUOp_or:   C = A | B;
`ALUOp_sll:  C = A << B[4:0];
`ALUOp_srl:  C = $unsigned(A) >> B[4:0];
```

分支条件：

```verilog
assign Zero = (ALUOp == `ALUOp_bne ||
               ALUOp == `ALUOp_blt ||
               ALUOp == `ALUOp_bge ||
               ALUOp == `ALUOp_bltu ||
               ALUOp == `ALUOp_bgeu) ? (C != 32'b0) : (C == 32'b0);
```

说明：

- 普通 `beq` 用减法结果是否为 0 判断。
- `bne/blt/bge/bltu/bgeu` 这类比较指令直接让 ALU 输出条件真假，`Zero` 表示分支是否成立。

### 4. 下一条 PC 生成

文件：`source-sc/NPC.v`

```verilog
case (NPCOp)
    `NPC_PLUS4:  NPC = PC + 4;
    `NPC_BRANCH: NPC = PC + IMM;
    `NPC_JUMP:   NPC = PC + IMM;
    `NPC_JALR:   NPC = (RS1 + IMM) & 32'hffff_fffe;
    default:     NPC = PC + 4;
endcase
```

说明：

- 普通指令顺序执行，下一条地址为 `PC + 4`。
- B 型和 J 型立即数已经在 `EXT.v` 中左移一位，因此这里直接加 `IMM`。
- `jalr` 按 RISC-V 规范清零目标地址最低位。

### 5. 寄存器堆

文件：`source-sc/RF.v`

```verilog
if (RFWr) begin
    if (A3 != 5'b0) begin
        rf[A3] <= WD;
    end
end

assign RD1 = (A1 != 0) ? rf[A1] : 0;
assign RD2 = (A2 != 0) ? rf[A2] : 0;
```

说明：

- `x0` 永远读出 0。
- 写回时如果目的寄存器是 `x0`，写操作被忽略。

## 四、多周期 CPU 关键代码

文件：`source-mc/MCCPU.v`

### 1. 状态划分

```verilog
localparam S_FETCH  = 3'd0;
localparam S_DECODE = 3'd1;
localparam S_EXEC   = 3'd2;
localparam S_MEM_RD = 3'd3;
localparam S_MEM_WR = 3'd4;
localparam S_WB     = 3'd5;
```

每条指令不再一个周期完成，而是按阶段执行：

```text
取指 FETCH -> 译码 DECODE -> 执行 EXEC -> 访存 MEM -> 写回 WB
```

不同指令路径：

```text
R/I/lui : FETCH -> DECODE -> EXEC -> WB
lw      : FETCH -> DECODE -> EXEC -> MEM_RD -> WB
sw      : FETCH -> DECODE -> EXEC -> MEM_WR
branch  : FETCH -> DECODE -> EXEC
jal/jalr: FETCH -> DECODE -> EXEC -> WB
```

### 2. 多周期取指

```verilog
S_FETCH: begin
    ir    <= inst_in;
    pc    <= pc + 32'd4;
    state <= S_DECODE;
end
```

说明：

- `ir` 保存当前指令，后续周期都从 `ir` 译码。
- `pc` 在取指阶段先加 4，所以执行 `branch/jal` 时需要用 `pc - 4` 得到当前指令地址。

### 3. 多周期执行和跳转

```verilog
7'b1100011: begin // branch
    if (branch_taken(rd1, rd2, funct3))
        pc <= (pc - 32'd4) + imm_b;
    state <= S_FETCH;
end

7'b1101111: begin // jal
    aluout_reg <= pc;
    pc <= (pc - 32'd4) + imm_j;
    state <= S_WB;
end

7'b1100111: begin // jalr
    aluout_reg <= pc;
    pc <= (rd1 + imm_i) & 32'hffff_fffe;
    state <= S_WB;
end
```

说明：

- 因为取指阶段已经 `pc = pc + 4`，所以 `aluout_reg <= pc` 刚好保存返回地址。
- `branch/jal` 的目标地址要从当前指令地址算起，所以使用 `(pc - 4) + imm`。
- `jalr` 目标地址由寄存器加立即数得到，并清零最低位。

### 4. 多周期访存和写回

```verilog
7'b0000011: begin // lw
    addr_reg <= rd1 + imm_i;
    state <= S_MEM_RD;
end

7'b0100011: begin // sw
    addr_reg  <= rd1 + imm_s;
    wdata_reg <= rd2;
    state <= S_MEM_WR;
end

S_MEM_RD: begin
    mdr <= Data_in;
    state <= S_WB;
end

S_WB: begin
    wd_reg <= (op == 7'b0000011) ? mdr : aluout_reg;
    rf_we_reg <= 1'b1;
    state <= S_FETCH;
end
```

说明：

- `lw` 先计算地址，再读内存，再写回寄存器。
- `sw` 只需要计算地址和写数据，不进入写回阶段。
- `mdr` 保存内存读出的数据，避免跨周期数据丢失。

## 五、单周期、多周期与流水线对比

### 1. 总体区别

| 对比项 | 单周期 CPU | 多周期 CPU | 流水线 CPU |
|---|---|---|---|
| 指令执行方式 | 一条指令一个完整时钟周期执行完 | 一条指令拆成多个状态分多拍执行 | 多条指令在不同阶段重叠执行 |
| 典型阶段 | 一个周期内包含 IF/ID/EX/MEM/WB | FETCH、DECODE、EXEC、MEM、WB 分状态 | IF、ID、EX、MEM、WB 同时并行推进 |
| 时钟周期 | 必须按最慢指令设置，周期较长 | 每拍只做一部分工作，周期可缩短 | 每拍也只做一个阶段，吞吐率高 |
| CPI | 理想情况下 CPI=1 | CPI 大于 1，不同指令不同 | 理想情况下 CPI 接近 1 |
| 硬件复用 | 同一周期内资源不能跨阶段复用，组合路径长 | ALU、存储器等资源可跨周期复用 | 需要流水线寄存器和冒险处理逻辑 |
| 控制复杂度 | 最简单 | 中等，需要状态机 | 最复杂，需要暂停、转发、冲刷 |
| 本实验对应文件 | `source-sc/SCCPU.v` | `source-mc/MCCPU.v` | `source-pipeline/PipelineCPU.v` |

验收时可以这样概括：

```text
单周期关注“每条指令一拍完成”，多周期关注“把一条指令拆阶段复用硬件”，流水线关注“多条指令阶段重叠提高吞吐率”。
```

### 2. 单周期和多周期的关键不同

单周期 CPU 的核心特点：

- `PC`、`im`、`ctrl`、`RF`、`EXT`、`ALU`、`dm`、`NPC` 在一个周期内串起来。
- 指令从取指到写回都在同一拍完成。
- 控制器是组合逻辑，直接根据当前指令生成控制信号。
- 写回数据通过 `WDSel` 在 `ALU/MEM/PC+4` 中选择。

多周期 CPU 的核心特点：

- 使用 `state` 状态机控制执行流程。
- 使用 `ir` 保存当前指令，避免下一拍 PC 改变后指令丢失。
- 使用 `aluout_reg`、`addr_reg`、`wdata_reg`、`mdr` 等寄存器保存跨周期中间结果。
- 不同指令执行周期数不同，例如 `sw` 不需要写回，`branch` 不需要访存和写回。

### 3. 为什么多周期 CPU 需要额外寄存器？

因为一条指令不在一个时钟周期内完成。每个阶段结束后，下一阶段还要继续使用前一阶段的结果，所以必须把中间结果保存下来。

常见寄存器作用：

| 寄存器 | 作用 |
|---|---|
| `ir` | 保存当前指令，供后续周期译码和执行 |
| `pc` | 保存当前或下一条指令地址 |
| `aluout_reg` | 保存 ALU 结果或 `jal/jalr` 返回地址 |
| `addr_reg` | 保存 `lw/sw` 访存地址 |
| `wdata_reg` | 保存 `sw` 要写入内存的数据 |
| `mdr` | 保存 `lw` 从内存读出的数据 |
| `rd_reg` | 保存目的寄存器编号 |

### 4. 单周期、多周期和流水线的性能理解

假设一条指令最多包含 5 个阶段：

```text
IF  取指
ID  译码/读寄存器
EX  执行/地址计算
MEM 访存
WB  写回
```

单周期：

```text
一条指令必须在一个很长的周期内完成 IF+ID+EX+MEM+WB。
```

多周期：

```text
一条指令分 3 到 5 个较短周期完成，不同指令可以少走不需要的阶段。
```

流水线：

```text
第 1 条指令执行 ID 时，第 2 条指令可以同时执行 IF；
第 1 条指令执行 EX 时，第 2 条指令执行 ID，第 3 条指令执行 IF。
```

流水线不能明显降低单条指令的延迟，但可以提高连续指令的吞吐率。

### 5. 本实验流水线 CPU 结构

文件：`source-pipeline/PipelineCPU.v`

本实验新增的流水线 CPU 采用五级流水线：

```text
IF  -> ID -> EX -> MEM -> WB
取指   译码   执行   访存   写回
```

关键流水线寄存器：

| 流水线寄存器 | 作用 |
|---|---|
| `IF/ID` | 保存取出的指令和对应 PC |
| `ID/EX` | 保存译码结果、寄存器读数、立即数和控制信号 |
| `EX/MEM` | 保存 ALU 结果、store 写数据、访存控制信号 |
| `MEM/WB` | 保存内存读数或 ALU 结果，供写回阶段使用 |

流水线顶层连接：

```text
pipecomp.v
  |-- PipelineCPU.v
  |-- im.v
  |-- dm.v
```

与多周期 CPU 的关键不同：

- 多周期用 `state` 控制同一条指令分阶段执行。
- 流水线没有全局“当前指令状态机”，而是每拍推进多条指令。
- 多周期使用 `ir/mdr/aluout_reg` 保存单条指令的中间结果。
- 流水线使用 `IF/ID`、`ID/EX`、`EX/MEM`、`MEM/WB` 保存不同指令在不同阶段的信息。
- 流水线必须增加 `hazard_unit.v` 处理冒险，多周期不需要。

## 六、流水线冒险及解决方法

流水线冒险是指多条指令重叠执行时，后一条指令因为依赖、资源冲突或跳转而不能按理想情况继续推进。常见分为结构冒险、数据冒险、控制冒险。

### 1. 结构冒险

含义：

```text
多个流水线阶段在同一周期竞争同一个硬件资源。
```

例子：

```text
IF 阶段需要读指令存储器，同时 MEM 阶段需要读/写数据存储器。
如果指令和数据共用一个单端口存储器，就会冲突。
```

解决方法：

- 指令存储器和数据存储器分离，即哈佛结构。
- 使用双端口存储器。
- 如果硬件资源无法增加，就插入暂停周期 `stall`。

本实验中 `im.v` 和 `dm.v` 是分开的，所以从结构上避免了取指和访存冲突。

### 2. 数据冒险

含义：

```text
后一条指令需要使用前一条指令的结果，但前一条指令还没有写回寄存器。
```

典型例子：

```asm
add x1, x2, x3
sub x4, x1, x5
```

`sub` 在 ID/EX 阶段就要读 `x1`，但 `add` 的结果可能还没有到 WB 阶段。

常见解决方法：

1. 数据转发 `forwarding/bypassing`

   不等结果写回寄存器，直接从 EX/MEM 或 MEM/WB 流水线寄存器把结果送回 ALU 输入端。

   典型判断：

   ```text
   如果 EX/MEM.rd == ID/EX.rs1，并且 EX/MEM.RegWrite=1，则 rs1 从 EX/MEM 转发。
   如果 MEM/WB.rd == ID/EX.rs2，并且 MEM/WB.RegWrite=1，则 rs2 从 MEM/WB 转发。
   ```

2. 暂停 `stall`

   对无法立即转发的情况，暂停后续指令，让前一条指令先产生结果。

   最典型是 `lw-use` 冒险：

   ```asm
   lw  x1, 0(x2)
   add x3, x1, x4
   ```

   `lw` 的数据要到 MEM 阶段结束才有效，下一条 `add` 的 EX 阶段太早，即使转发也来不及，通常需要插入 1 个气泡。

3. 编译器或程序手动重排

   在有依赖的两条指令之间插入无关指令，减少硬件暂停。

### 3. 控制冒险

含义：

```text
分支或跳转指令会改变 PC，但流水线已经提前取了后续顺序指令。
```

例子：

```asm
beq x1, x2, target
addi x3, x0, 1
```

如果 `beq` 成立，`addi` 可能已经被取进流水线，但它不应该执行。

解决方法：

- 等分支结果确定后再取下一条指令，代价是暂停。
- 分支判断尽量提前，例如放到 ID 或 EX 阶段。
- 冲刷流水线 `flush`，把已经错误取入的指令变成空操作 `nop`。
- 使用分支预测，预测失败时再冲刷错误路径。

对本实验排序程序来说，循环和 `jal/jalr` 较多。流水线 CPU 执行这些分支、跳转和 `swap` 子程序调用时，会产生明显控制冒险，因此需要对错误路径指令进行冲刷。

### 4. 验收时如何回答“流水线冒险怎么解决？”

可以按三句话回答：

```text
结构冒险靠资源分离或暂停解决，比如指令存储器和数据存储器分开。
数据冒险优先用转发解决，load-use 这类转发来不及的情况插入 stall。
控制冒险靠提前判断分支、分支预测或 flush 冲刷流水线解决。
```

如果老师继续追问本实验如何迁移到流水线，可以回答：

```text
本实验已经在 source-pipeline 中加入五级流水线 CPU。
PipelineCPU.v 中包含 IF/ID、ID/EX、EX/MEM、MEM/WB 流水线寄存器。
hazard_unit.v 负责 load-use 暂停、EX/MEM 和 MEM/WB 转发、branch/jal/jalr 冲刷。
```

### 5. 单周期和多周期为什么没有流水线冒险？

单周期 CPU 一次只完成一条指令，下一条指令要等当前指令完全结束后才开始，不存在多条指令重叠执行。

多周期 CPU 虽然一条指令拆成多个阶段，但同一时间仍然只处理一条指令，也不存在后一条指令提前读取前一条结果的问题。

流水线 CPU 才会让多条指令同时处在不同阶段，因此才需要专门处理冒险。

## 七、排序程序说明

文件：`source-sc/rv32_sid_sort_sim.asm` 和 `source-mc/rv32_sid_sort_sim.asm`

排序程序把 8 位学号按每 4 bit 一个 BCD 数字处理。核心思想可以这样讲：

1. 用移位和加法构造原始 BCD 学号 `0x54873530`。
2. 把原始学号写入 `0x180`，用于验收检查。
3. 外层循环控制当前待放置的位置。
4. 内层循环扫描剩余 BCD 数字，通过掩码和移位取出单个数字。
5. 使用 `slt` 比较大小，必要时调用 `swap` 子程序交换两个 4 bit 数字。
6. 排序完成后写入 `0x184`，结果为 `0x03345578`。

容易被问到的一点：虽然程序从低位 nibble 开始处理，最终 32 位十六进制数从高位到低位读出来就是升序结果 `0,3,3,4,5,5,7,8`。

## 十、可能的验收问题

### 1. 你的实验实现了什么？

实现了 RV32I 子集 CPU，能够执行学号 BCD 排序程序。工程中包括单周期 CPU、多周期 CPU、五级流水线 CPU、仿真 testbench 和 Nexys A7 上板顶层。仿真检查原始学号 `0x54873530` 和排序结果 `0x03345578`。

### 2. 单周期 CPU 的特点是什么？

单周期 CPU 每条指令在一个时钟周期内完成取指、译码、执行、访存和写回。优点是控制简单，缺点是时钟周期必须满足最慢指令的组合逻辑延迟。

### 3. 多周期 CPU 相比单周期 CPU 改了什么？

多周期 CPU 把一条指令拆成多个阶段，用状态机控制 `FETCH/DECODE/EXEC/MEM/WB`。不同指令可以走不同长度的路径，例如 `sw` 不需要写回，`branch` 不需要访存和写回。

### 4. 单周期、多周期、流水线三者最大的区别是什么？

单周期是一条指令一拍完成，控制简单但周期长；多周期是一条指令分多拍完成，能复用硬件并缩短单拍延迟；流水线是多条指令重叠执行，单条指令仍经过多个阶段，但整体吞吐率更高。

### 5. 多周期 CPU 为什么要有 `ir`、`mdr` 这些寄存器？

因为一条指令跨多个时钟周期执行，中间结果不能只靠组合线保存。`ir` 保存当前指令，`mdr` 保存内存读数，`aluout_reg` 保存 ALU 结果或返回地址，保证下一状态还能继续使用。

### 6. 单周期和多周期有没有流水线冒险？

没有。单周期和多周期同一时间只执行一条指令，不存在多条指令重叠造成的资源冲突、数据依赖提前读取或错误路径取指。流水线让多条指令重叠执行后，才需要专门处理冒险。

### 7. 流水线有哪些冒险？怎么解决？

结构冒险：多个阶段抢同一硬件资源，靠指令/数据存储器分离、双端口资源或暂停解决。

数据冒险：后一条指令要用前一条还没写回的结果，优先用转发解决；`lw-use` 这类来不及转发的情况插入 `stall`。

控制冒险：分支或跳转改变 PC，但顺序指令已被取入，靠提前判断、分支预测或 `flush` 冲刷错误指令解决。

### 8. 什么是 `lw-use` 冒险？

例如：

```asm
lw  x1, 0(x2)
add x3, x1, x4
```

`add` 立刻使用 `lw` 读出的 `x1`，但 `lw` 的数据要到 MEM 阶段结束才有效，下一条指令的 EX 阶段来不及拿到数据，通常需要暂停 1 个周期再转发。

### 9. 本实验流水线 CPU 增加了哪些部件？

`source-pipeline/PipelineCPU.v` 中增加了 IF/ID、ID/EX、EX/MEM、MEM/WB 流水线寄存器。`pipe_decode.v` 负责译码，`pipe_alu.v` 负责 EX 阶段运算，`pipe_branch.v` 负责分支和跳转目标，`source-pipeline/hazard_unit.v` 负责 forwarding、load-use 检测和 flush 控制。流水线仿真顶层为 `source-pipeline/pipecomp.v`。

### 10. `hazard_unit.v` 具体做什么？

它做三件事：

- 检测 `lw-use` 冒险，产生 `pc_write=0`、`ifid_write=0`，并向 `ID/EX` 插入 bubble。
- 检测 `EX/MEM` 和 `MEM/WB` 是否可以转发数据到 EX 阶段，输出 `forward_a` 和 `forward_b`。
- 当 EX 阶段发现 `branch/jal/jalr` 改变 PC 时，产生 flush 信号，清掉错误取入的指令。

### 11. 为什么 `0x180` 对应 `dmem[96]`？

数据存储器按 32 位字访问，地址低两位用于字节偏移，不参与字索引。代码使用 `addr[8:2]` 作为数组下标，所以 `0x180 >> 2 = 0x60 = 96`。

### 12. `jal` 和 `jalr` 为什么要写回 `PC + 4`？

RISC-V 规定跳转链接指令把下一条指令地址写入 `rd`，作为函数返回地址。程序中调用 `swap` 子程序时，需要用这个返回地址跳回调用点之后继续执行。

### 13. `jalr` 为什么要清零最低位？

RISC-V 规范要求 `jalr` 的目标地址最低位清零，即 `(rs1 + imm) & ~1`。代码中实现为：

```verilog
NPC = (RS1 + IMM) & 32'hffff_fffe;
```

### 14. `lw` 和 `sw` 的控制信号有什么区别？

`lw`：需要 `RegWrite=1`，`ALUSrc=1`，ALU 做加法计算地址，写回来源选择内存。

`sw`：需要 `MemWrite=1`，`ALUSrc=1`，ALU 做加法计算地址，不写寄存器。

### 15. 立即数扩展模块支持哪些类型？

支持 I、S、B、U、J 五类立即数。B 型和 J 型跳转地址最低位固定为 0，因此扩展时会在末尾补 `1'b0`，等价于左移一位。

### 16. 为什么寄存器 `x0` 不能被写入？

RISC-V 规定 `x0` 恒为 0。寄存器堆中写回时判断 `A3 != 0`，读出时如果地址为 0 就直接返回 0。

### 17. ALU 中 `Zero` 信号一定表示结果等于 0 吗？

不是。对普通比较如 `beq`，`Zero` 表示减法结果为 0。对 `bne/blt/bge/bltu/bgeu`，ALU 输出的是条件真假，此时 `Zero` 实际表示分支条件是否成立。

### 18. 排序结果为什么是 `03345578`？

原始学号数字为 `5,4,8,7,3,5,3,0`，升序排列后为 `0,3,3,4,5,5,7,8`，BCD 十六进制显示为 `0x03345578`。


这些指令覆盖了数据构造、循环、比较、移位提取 BCD、访存和子程序调用。
